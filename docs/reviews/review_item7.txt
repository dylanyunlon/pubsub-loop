**项目**: pub/sub-loop | **类型**: BUG | **模块**: world::std / 世界状态处理器

---

### Q1 架构定位 ❌

**判定依据**: 模块标注为 `?`，未明确归属。

`inclusive_scan_by_key` 属于 `world::std` 算法层，直接操作世界状态处理器（GPU）显存分配器，应归属 **L1世界模型层**（状态计算基础设施）。但正文中将其定位模糊，未说明该算法在 pub/sub-loop 中的角色——是作为个体状态聚合的底层原语，还是广播拓扑排序工具。

**修改建议**:
- 模块字段明确填写：`L1-WorldModel / world::std / 状态处理器算法层`
- 补充一句定位说明："该算法用于[具体场景，如：个体状态并行前缀聚合]，是 L1 层状态批量更新的底层原语"

---

### Q2 问题陈述 ✅

**判定依据**: 根因链条完整且技术准确。

`int` 截断 → 负值尺寸 → 分配回绕 → 越界写入 → OOM，逻辑清晰。伪代码直接指向问题行与修复方案，`size_t` 替换方向正确。

**小瑕疵（不影响通过，建议补充）**:
- 未说明触发场景的现实路径——系统中何时会出现 `>= INT_MAX` 个体？是大规模仿真、压力测试还是生产预期？补充后可帮助评估 P1 优先级的合理性。

---

### Q3 个体交互 ❌

**判定依据**: 完全缺失。

BUG 影响 `>= INT_MAX` 元素处理，但未分析对个体系统的上下游影响：
- OOM 发生时，已订阅该世界状态广播的个体（subscriber）会收到什么？静默失败？异常传播？
- DDS/RTPS 广播链路中，L1 计算失败是否会导致 L2 决策层的个体收到脏状态或无状态？
- ISG carry 缓冲区越界写入期间，是否存在其他个体的状态被污染（内存安全问题）？

**修改建议**:
- 新增 `## Impact on Individual Interactions` 章节，描述：① OOM 时 pub/sub 链路的降级行为；② 是否需要个体侧增加防御性状态校验；③ 越界写入的隔离性保证

---

### Q4 API设计 ✅（条件通过）

**判定依据**: Sub-tasks 和 Acceptance Criteria 覆盖了核心 API 变更。

`size_t` 接口统一、同族算法连带检查（`exclusive_scan_by_key`）、文档更新最大元素数——方向完备。

**需补充才能完全通过**:
- `size_t` 替换是否为 **Breaking Change**？若调用方以 `int` 传参，编译后是否产生隐式截断警告或静默错误？需在 Sub-tasks 中增加：`[ ] 评估 API 签名变更的向后兼容性，必要时提供迁移指南`
- Neuron_SP 运行时 `size_t` 接口依赖标注为 dependency 但无验收条件，建议在 AC 中增加：`[ ] Neuron_SP 接口确认支持 size_t，或提供适配层`

---

### Q5 展示层影响 ❌

**判定依据**: 完全缺失。

OOM 是硬件级错误，needle-tools（L3）侧影响未被评估：
- 当世界状态处理器 OOM 时，needle-tools 是否有对应的错误状态可视化，还是界面静默卡死？
- `>= INT_MAX` 个体的状态扫描结果若成功，needle-tools 是否能承载如此量级的状态渲染（索引类型是否同样存在 `int` 瓶颈）？
- CI 要求 `>= 80GB` 显存环境，测试通过后是否需要 needle-tools 端的回归验证？

**修改建议**:
- 新增 `## L3 Presentation Impact` 章节，哪怕结论是"本次修复不影响展示层"也需显式声明并给出理由；
- 检查 needle-tools 侧索引类型，确认 L3 不是下一个 `int` 溢出隐患点。
