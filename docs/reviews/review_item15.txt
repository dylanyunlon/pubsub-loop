### Q4 API设计 ❌

**判定理由：** 对于一个BUG ticket，出现完整的新API设计（`Config`结构体 + `Processor`类）是严重的范围蔓延。且API本身存在设计缺陷。

**具体问题：**

| 问题 | 位置 | 说明 |
|---|---|---|
| 范围越界 | 整个API章节 | BUG单不应引入新公共API |
| 错误处理缺失 | `initialize()` 返回`bool` | 应返回`Result<void, Error>`或抛出，bool无法携带错误信息 |
| 线程安全未声明 | `Processor`类 | pub/sub多个体并发调用时是否需要外部加锁？ |
| `from_json`异常策略 | `Config::from_json` | JSON解析失败时行为未定义 |
| 生命周期未说明 | `StateView` | 是否是`StateBuffer`的视图？`StateBuffer`析构后是否悬空？ |

**修改建议：**
```
若必须保留API章节（定性为Feature），修改如下：

// 错误处理
std::expected<void, ToolsError> initialize();

// 线程安全注释
/// @thread-safety: NOT thread-safe. External synchronization required.
StateView process_tick(const StateBuffer& s, uint64_t tick);

// 生命周期明确
/// @note: StateView lifetime is bounded by the StateBuffer argument.
StateView process_tick(const StateBuffer& s, uint64_t tick);

若保持BUG ticket定位，删除此章节，仅保留测试用例代码。
```
