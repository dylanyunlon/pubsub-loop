## pub/sub-loop PRD 准入审核报告

**工单**: [BUG] Consolidate conflicting licenses | P1 🟡

---

### Q1 架构定位 ❌

**判定**: 不合格

**问题**: 工单未标注受影响模块（填写 `?`），且正文中对三层架构的定位完全缺失——没有说明此变更属于 L1/L2/L3 哪一层，或跨层。

**修改建议**:
- 模块字段改为 `common / 全仓库（跨层静态文件）`
- 正文开头补充一句架构定位声明，例如：

> 本工单属于**基础设施层（Infrastructure）**静态变更，不隶属 L1/L2/L3 运行时架构，但影响所有三层模块的文件合规性。

---

### Q2 问题陈述 ✅

**判定**: 合格（有条件）

**优点**: 描述了三类冲突场景（文件头 vs 根 LICENSE、GPL 污染、SPDX 不一致），后果（CI 报错、合规阻断）清晰，验收标准可量化。

**轻微缺陷**: "存在的具体冲突类型（需人工确认）"——提交时冲突尚未扫描确认，属于假设性问题陈述，风险是修复范围可能与实际严重偏离。

**建议**: 将 `licensee` / `scancode` 的实际扫描输出作为**前置附件**附上后再提交，或将优先级暂降为 P2 草稿态，扫描确认后升级。

---

### Q3 个体交互 ✅

**判定**: 合格

**理由**: 工单已明确声明：

> 不涉及任何个体（Individual）运行时行为；不涉及 pub/sub 通道、Cyber RT、传输层。

这是此类工程合规票的**正确写法**——显式排除比不提更好，审核者可快速确认无个体状态机、消息路由变更。

**无需修改。**

---

### Q4 API 设计 ❌

**判定**: 不合格

**问题**: 工单注明"无运行时 API 变更"，但缺少对以下内容的显式说明：

1. `common/include/*.h` 中的**公开头文件**若修改了文件头注释块，是否会触发下游构建系统的 hash/fingerprint 变更？
2. 若替换第三方依赖（`/third_party/`），依赖的 **ABI / 接口签名**是否保持兼容？
3. CI `license-check job` 的新增——其失败策略（block merge vs warn-only）属于**隐式 API 契约变更**，影响所有 PR 提交流程。

**修改建议**: 在"对其他模块的影响"表格中补充一列：

| 影响项 | 说明 | **API/接口兼容性** |
|--------|------|---|
| `common` 模块 | 主要修改区域 | **仅注释变更，ABI 不变（需确认）** |
| 第三方依赖 | 可能需替换 | **替换前需做 ABI 兼容性验证** |
| CI pipeline | 新增 license-check | **阻断策略需在此明确（block/warn）** |

---

### Q5 展示层影响 ❌

**判定**: 不合格

**问题**: 工单完全未提及 **L3 展示层（needle-tools）**。虽然许可证变更是静态文件操作，但存在以下遗漏风险：

1. `needle-tools` 若作为**独立发布包**（npm / PyPI / 二进制分发），其 `package.json` / `setup.py` / `Cargo.toml` 中的 `license` 字段是否同步更新？
2. needle-tools 的用户文档、README 中若有许可证徽章（badge）或声明，需同步修改。
3. 若 needle-tools 集成了与主项目不同许可证的前端依赖（如 LGPL 的 UI 组件），GPL 污染分析需单独覆盖展示层。

**修改建议**: 在"涉及范围"下补充：

> **L3 needle-tools**：需确认独立分发包的 `license` 字段及文档徽章是否与根 LICENSE 一致；前端依赖需单独做许可证兼容性扫描。
